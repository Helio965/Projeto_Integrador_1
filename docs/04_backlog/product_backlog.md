# Product Backlog — NEXO

Histórias de usuário priorizadas pela técnica MoSCoW (Must / Should / Could / Won't).

## Must Have

| ID | História | Requisito |
|---|---|---|
| PB01 | Como ADMIN, quero acessar o sistema com login para garantir acesso restrito. | RF01, RF02 |
| PB02 | Como ADMIN, quero cadastrar uma empresa cliente para organizar as análises por cliente. | RF03 |
| PB03 | Como ADMIN, quero cadastrar um usuário CLIENTE vinculado à empresa para que ele acesse suas análises. | RF04 |
| PB04 | Como ADMIN, quero definir o plano da empresa para controlar a periodicidade das análises. | RF06, RF07 |
| PB05 | Como ADMIN, quero criar uma análise por período para consolidar os dados daquele intervalo. | RF08 |
| PB06 | Como ADMIN, quero fazer upload dos relatórios de vendas e compras para alimentar a análise. | RF09, RF10 |
| PB07 | Como ADMIN, quero que o sistema processe os relatórios e calcule os indicadores para obter resultados consolidados. | RF11, RF12 |
| PB08 | Como CLIENTE, quero visualizar o dashboard da análise para entender o desempenho do período. | RF13, RF15 |
| PB09 | Como ADMIN, quero registrar a devolutiva estratégica para orientar o cliente. | RF14 |

## Should Have

| ID | História | Requisito |
|---|---|---|
| PB10 | Como ADMIN, quero associar um segmento à empresa para contextualizar a análise. | RF05 |
| PB11 | Como CLIENTE, quero consultar o histórico de análises publicadas para acompanhar a evolução do negócio. | RF16 |

## Could Have

| ID | História | Observação |
|---|---|---|
| PB12 | Como CLIENTE, quero destacar visualmente os produtos com maior saldo estimado parado. | Depende de dados suficientes nos relatórios. |

## Won't Have (neste ciclo)

- Avaliação da análise pelo cliente.
- Cobrança real, faturas e cliente vencido.
- Integração automática com PDV via API.
- Aplicativo mobile nativo.
- Machine learning ou análise preditiva.

> O status real de cada item deve refletir o quadro do GitHub Projects/Trello da equipe.
